# Description
The `SnowmanAirdrop` contract utilizes EIP-712 typed data signatures to verify that a claim is authorized by the recipient. This process depends on a constant `MESSAGE_TYPEHASH` that must exactly match the string representation of the signed struct.

However, there is a critical typo in the `MESSAGE_TYPEHASH` declaration. The word `address` is incorrectly spelled as `addres` (missing the final "s").

```solidity
// src/SnowmanAirdrop.sol

    // @> Vulnerability: Typo in the type string ('addres' instead of 'address')
    bytes32 private constant MESSAGE_TYPEHASH = keccak256("SnowmanClaim(addres receiver, uint256 amount)");
```

Because off-chain libraries (like Ethers.js or Viem) and user wallets will always use the correct spelling of the `address` type when generating signatures, the cryptographic digest calculated by the contract will never match the digest signed by the user.

# Risk
## Likelihood: High
The typo is hardcoded in a `private constant`. Since EIP-712 requires an exact character-for-character match of the type string, every legitimate signature generated by standard tools will fail the verification check in the smart contract.

## Impact: High
The signature verification mechanism is completely broken. This prevents any user from successfully using the delegated claim functionality (`claimSnowman` via signature), which is a core feature of the airdrop protocol.

# Proof of Concept
The following Proof of Concept compares the contract's incorrect typehash with the standard-compliant typehash that a frontend or wallet would use. The test confirms that they do not match, which leads to a permanent validation failure.

```solidity
function testFinding2_TypehashTypo() public {
    // The correct typehash string that wallets/frontend will use
    bytes32 CORRECT_TYPEHASH = keccak256("SnowmanClaim(address receiver, uint256 amount)");
    
    // The actual hash generated by the contract's misspelled constant
    bytes32 contractTypeHash = airdrop.getMessageHash(alice);
    
    // Assertion: They are not equal, meaning all signatures will be rejected
    assertFalse(
        CORRECT_TYPEHASH == contractTypeHash, 
        "Mismatch between standard and contract typehash"
    );
}
```

# Recommended Mitigation
Correct the spelling of the `address` type in the `MESSAGE_TYPEHASH` constant to ensure compatibility with standard EIP-712 signing tools.

```diff
// src/SnowmanAirdrop.sol

- bytes32 private constant MESSAGE_TYPEHASH = keccak256("SnowmanClaim(addres receiver, uint256 amount)");
+ bytes32 private constant MESSAGE_TYPEHASH = keccak256("SnowmanClaim(address receiver, uint256 amount)");
```
